# Waystation Codebase Audit — 2026-07-08

Full audit of ~6,800 LOC across the CLI, core ledger, SQLite index, MCP server,
and Hono/React dashboard.

**Baseline at audit time:** `tsc --noEmit` clean · 117 tests pass · Biome clean.

Each finding has a **Status**:
- **FIXED** — addressed in this pass (commit accompanying this doc).
- **OPEN** — documented, not yet fixed.

Severity is impact-first: a High finding is one that corrupts the ledger, leaks
data, or silently produces wrong coordination decisions.

---

## High priority

### H1 — 95 MB build artifact committed to git history — **FIXED (partial)**
`.18bfc7bffff7fdbe-00000000.bun-build` (99.8 MB) was tracked in `HEAD`; the packed
`.git` is 76 MB, dominated by this blob. `.gitignore` did not cover `*.bun-build`,
`stderr.txt`, or `stdout.txt`, so a second ~100 MB artifact was stage-able.

- **Fixed now:** `git rm --cached` the tracked blob and added ignore rules for
  `*.bun-build`, `stderr.txt`, `stdout.txt`.
- **Still OPEN (needs explicit go-ahead — history rewrite is destructive):** the
  95 MB blob remains in history and will bloat every clone until removed with
  `git filter-repo --path .18bfc7bffff7fdbe-00000000.bun-build --invert-paths`
  (or BFG), followed by a force-push. Do this before the repo is widely cloned.

### H2 — Dashboard has no auth / CSRF / Origin protection — **FIXED**
`src/dashboard/server.ts`. Binding to `127.0.0.1` is not a boundary against the
browser. While `waystation dashboard` runs, any page the user visits could
`fetch('http://127.0.0.1:PORT/api/git/commit', {method:'POST', body:'…'})`; the
body is sent as `text/plain` (a CORS "simple" request, no preflight) and
`c.req.json()` parses it regardless of Content-Type, so the **side effect
executes** even though the response is unreadable. DNS-rebinding defeats the
localhost binding entirely.

- **Fix:** a Hono middleware that (a) rejects any request whose `Host` header is
  not a loopback host (`localhost`/`127.0.0.1`/`::1`) — defeats DNS rebinding for
  both reads and writes; and (b) for mutating methods, rejects requests whose
  `Origin` host does not equal the `Host` — defeats cross-site simple-request
  CSRF. Non-browser clients (no `Origin`, loopback `Host`) are unaffected, so the
  CLI/tests keep working.

### H3 — GitHub export/import trusts a caller-supplied repo + injects it into URLs — **FIXED**
`src/dashboard/server.ts:180` → `src/core/gh.ts`. `/api/gh/export` POSTed every
ledger issue to an arbitrary `repo` using the user's `GITHUB_TOKEN`, with no
validation; `repo` was also spliced unescaped into `api.github.com` URLs
(`owner/name/../../user` → request-path injection on the authenticated call).

- **Fix:** `assertValidRepo()` rejects anything not matching a strict
  `^[A-Za-z0-9._-]+/[A-Za-z0-9._-]+$` before any network call, in both import and
  export (new `invalid_github_repo` diagnostic). Combined with H2, the
  drive-by-exfiltration path is closed. Pinning the export destination to a
  configured repo remains a possible hardening (OPEN, low urgency).

### H4 — Multi-file mutations are not crash-atomic — **FIXED (mitigated + detected)**
`src/core/mutate.ts`. `withLedgerLock` gives mutual exclusion but no transaction:
each mutation does 2–4 independent `writeJsonAtomic`/`appendEventUnlocked` calls.
A crash between `writeClaim` and `writeTask` leaves an `active` claim on a still-
`ready` task.

- **Fix (this pass):** (1) event log is now **write-ahead** — events are appended
  before record writes, so the audit log is always a superset of applied changes;
  (2) `validateLedger` now flags an active claim on a task that is not
  `in_progress` (new `claim_status_divergence` warning), so a partial mutation is
  *detectable*; (3) the rename-retry in H5 removes the most common trigger of a
  mid-mutation throw.
- **Still OPEN:** true all-or-nothing multi-file commit (stage every file for a
  mutation, then rename them as the last step, or a write-then-replay journal).
  This is the complete fix; the above is a strong mitigation + detection net.

### H5 — `renameSync` over an open file throws on Windows; readers take no lock — **FIXED**
`src/core/store.ts:56`. On Windows, `rename` onto a destination another handle has
open throws `EPERM`/`EBUSY`. The polling dashboard (and antivirus) hold read
handles exactly when a CLI write renames over the file, so writes intermittently
throw mid-mutation (compounding H4).

- **Fix:** `writeJsonAtomic` retries the rename with short backoff on
  `EPERM`/`EBUSY`/`EACCES` (the standard Windows atomic-write pattern).

### H6 — A `wont_do` dependency permanently, silently blocks all dependents — **FIXED**
`src/core/tasks.ts:13` and `src/index/taskIndex.ts:55`. Readiness only accepted
`done`; a dependency closed as `wont_do` (a legitimate terminal state) made the
whole chain behind it unactionable with no diagnostic.

- **Fix:** dependency satisfaction now accepts `done` **or** `wont_do`, in both
  the in-memory `isActionable` and the SQL `readyFromIndex` (`doneIds` set), so
  the two paths stay consistent.

### H7 — No concurrency / locking test — **FIXED**
The single write lock is the core safety property of a multi-agent tool, but the
"second claim rejected" test was fully sequential (it exercised the
`activeClaimForTask` guard, not the lock).

- **Fix:** added a test firing N concurrent `claimTask` calls at one task via
  `Promise.all` and asserting exactly one wins and the rest fail with
  `task_already_claimed`. Also added a `wont_do`-dependency readiness test and a
  `gh` repo-validation test.

## Low priority (OPEN)

- **Input/path hardening:** task `id` from a record body isn't charset-validated before becoming a filename (`../../evil` escapes `tasks/`) — `mutate.ts:30`; imported GitHub issue titles written unescaped into generated Markdown — `generate.ts:91`; `generate.ts` `writeText` uses a fixed `.tmp` name (races) — `generate.ts:19`.
- **ID collisions:** claim/handoff ids are second-resolution with no random suffix (unlike `postMessage`) — claim→release→re-claim within one second overwrites the released record — `mutate.ts:73`, `handoff.ts:42`.
- **Graph robustness:** empty-string path hint matches every node (`includes("")`) and floods the brief; corrupt `graph.json` degrades to empty-ok with no warning — `graph.ts:153`.
- **Client polish:** unencoded values in URLs (`?thread=${t}`), unawaited `clipboard.writeText`, fire-and-forget mutations without `r.ok` checks, un-sandboxed GraphView iframe.
- **CLI:** `--dev` proxy forwards `content-encoding`/`content-length` on an already-decoded body and throws if Vite isn't up yet — `cli/index.ts:623`; `process.exit(1)` after `stdout.write` can truncate piped `--json` output — `cli/index.ts:520`; `brief` mislabels unexpected errors as `no_such_task`.
- **Config:** Biome doesn't lint/format `.tsx` client code (`includes` omits `src/**/*.tsx`) — `biome.json`; `engines.node ">=22"` vs `@types/node ^26` skew; `/api/git/diff` ignores git exit codes.

## Confirmed-good (checked, not findings)

- All git/gh calls use array-form `Bun.spawnSync` (no shell) with `--` separators — no command injection.
- The user-facing id→filename path in `createIssue` is properly sanitized (charset + `resolve` + containment).
- No `dangerouslySetInnerHTML` anywhere; React escapes all record strings.
- Dashboard binds to `127.0.0.1`, not `0.0.0.0`.
- `withLedgerLock` is not re-entrant, but no locked path calls the self-locking `appendEvent` — no deadlock.
- `findCycle` handles self-dependency and the empty ledger correctly.
- The SQLite index is always rebuilt from JSON on demand — no stale-index divergence path today.
